# lark-github-sync (AI Agent Orchestrator)

## 目标

作为项目“大总管”，主动帮你同步 GitHub Issues 和飞书文档（Lark Docs）中的任务，整理成清晰的候选任务清单，并等待用户选择后再进入调研或执行。

**默认行为：只同步、排序、汇报任务；不要自动调研、不要自动执行、不要把调研结果标记为任务完成。**

## 触发时机

当用户输入 `/lark-github-sync` 或表达“帮我同步一下任务”、“看看今天有什么要做的”时触发。

## 执行流程 (SOP)

### 阶段一：任务收集与汇报

1.  **收集 GitHub 任务**：
    *   利用内部工具或 `gh` CLI 命令，查找当前项目分配给你的或者带有 `todo` 或 `doing` 标签的 Open Issues。
    *   对于 `doing` 或类似“进行中”状态的任务，优先识别当前是否已有对应的工作分支。
    *   *建议命令*：`gh issue list --state open --json number,title,labels,url`，过滤出标签中包含 todo、doing 或 in-progress 的 Issue。
2.  **收集飞书任务**：
    *   首先询问用户想要检查哪个飞书文档的 URL 或 Token（如果你还没有的话）。
    *   使用 `lark-doc` 技能，读取文档的所有 Block。
    *   筛选出 Block 类型为 Todo 且未完成（`done: false`）的项目。
3.  **生成任务清单**：
    *   向用户展示一个整合好的任务清单，必须包含：编号、来源、标题、优先级、状态、负责人、链接/记录 ID、是否需要用户补充上下文。
    *   按优先级排序：P0 > P1 > P2 > P3 > 无优先级；同优先级下“进行中/doing”优先于“未开始/todo”。
    *   对于仅有标题、缺少仓库/文档/验收标准的任务，必须标注：`需要用户提供上下文`，不要自行猜测。
    *   使用 `todowrite` 工具只记录当前同步流程，例如：`同步任务清单`、`等待用户选择任务`。**不要把每个飞书/GitHub 任务都加入 Todo 列表，更不要把它们标记为完成。**
    *   最后必须停止并询问用户选择任务编号，例如：“请选择要处理的任务编号；如任务需要外部仓库、PRD、飞书文档或验收标准，请一并提供。”

### 阶段一结束条件（硬性要求）

完成任务清单汇报后必须停止。除非用户已经在同一句话中明确指定了任务编号和处理方式，否则禁止进入调研或执行。

禁止：

*   不经用户选择就批量调研所有任务。
*   不经用户选择就按顺序执行所有任务。
*   将“已调研/已出方案”写成“任务已完成”。
*   在代码库或文档缺失时自行脑补业务背景。

允许：

*   对任务标题做轻量归类和优先级排序。
*   标注某任务缺少必要上下文。
*   如果用户已经明确指定任务编号，则直接进入阶段二。

### 阶段二：用户选择后的上下文确认与调研

当用户明确选择一个或多个任务编号后，先判断该任务是否具备执行所需上下文。

1.  **上下文检查**：
    *   明确任务来源、原始标题、记录/Issue 链接、当前状态、优先级。
    *   判断是否需要用户提供额外信息：代码仓库、分支、PRD/设计文档、测试范围、验收标准、目标环境、账号权限等。
    *   如果上下文缺失且无法从已提供链接中读取，必须向用户一次性列出缺失项；不要在当前仓库中无意义搜索。
2.  **调研任务标记规则**：
    *   如果用户只要求“看看/分析/调研/给方案”，则 Todo 名称必须写成 `调研：<任务标题>` 或 `方案：<任务标题>`。
    *   调研完成只能汇报为“调研完成/方案完成”，不能说“任务完成”。
    *   调研输出必须明确：已确认事实、未确认假设、需要用户补充的信息、可执行下一步。
3.  **执行前确认**：
    *   如果调研后要进入代码修改、创建 PR、更新飞书任务状态、打勾 Todo 等写操作，必须有用户明确授权或明确任务指令。
    *   用户只说“有哪些 todo / 同步一下 / 看看今天做什么”时，不构成执行授权。

### 阶段三：任务执行

当用户确认了要执行的具体任务后，按以下步骤处理每一个任务：

1.  **上下文准备**：
    *   如果任务来自 GitHub Issue：
        *   阅读完整的 Issue body (`gh issue view <id>`) 获取详细需求。
        *   如果是进行中 (`doing`) 的任务，先尝试查找本地或远程是否有对应分支 (`git branch -a | grep issue-<id>`)，如果有则切过去并拉取最新代码继续工作 (`git checkout <branch> && git pull`)。
        *   如果是全新任务 (`todo`)，则从主分支切出一个新的 feature 分支 (`git checkout -b feature/issue-<id>`)。
    *   如果任务来自飞书文档：解析 Todo 项的文本作为需求。如果只有标题或一句话，不能直接执行；必须要求用户提供对应代码仓库、文档、验收标准或更多背景。
2.  **规划与执行**：
    *   像平常一样，规划任务的实现步骤，调用各种编辑和搜索工具修改代码。
    *   执行必要的测试和验证。
3.  **提交与同步**：
    *   **代码提交**：只有用户明确要求提交时才 commit；需要提交时将更改提交 (`git commit -m "..."`) 并按需推送到远程分支。
    *   **如果是 GitHub 任务**：用户明确要求创建 PR 时，使用 `gh pr create` 创建 PR，并在描述中写上 `Resolves #<id>`。
    *   **如果是飞书任务**：只有在代码/文档实际交付并经验证后，才可将原任务标记为完成；不能因为调研完成就打勾。

## 约束与注意事项

*   **不要瞎猜上下文**：如果飞书里的 Todo 只有一行字（比如“修复登录页的 bug”），一定要结合当前项目代码去理解。如果不清楚，主动向用户提问。
*   **分支隔离**：建议为每个独立的任务创建一个新的分支，不要把所有任务揉在一个提交里，除非用户明确要求。
*   **尊重用户的决定**：如果用户说“今天只做第一个”，那就只做第一个，不要擅自行动。
*   **同步不是执行**：`/lark-github-sync` 默认只做任务发现和排序；用户选择任务编号前不得启动大规模调研或实现。
*   **调研不是完成**：调研、方案、代码修改、测试验证、PR/交付、飞书打勾是不同阶段，输出和 Todo 名称必须明确区分。
*   **缺上下文就停**：对应代码仓库、GitHub Issue、PRD、测试文档、验收标准不存在或用户未提供时，列出缺失项并等待用户补充，不要自行脑补。

## 依赖的内置技能

请确保在执行此 Skill 时，你已经了解如何使用以下技能：
*   `lark-doc` (处理飞书文档)
*   `git-master` 或基础的 git 终端命令 (处理分支和提交)
*   `gh` CLI 工具 (处理 GitHub Issues 和 PR)
